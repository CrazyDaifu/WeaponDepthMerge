项目所有者已经在 Battlefield 2 原生 D3D9 中实际测试 1.0，深度合并功能成功。1.1 RC1 已经实测解决 Battlefield 2 + DXVK 的启动闪退。RC2 解决了 Alt+Tab 返回后画面明显变暗的问题，但仍会卡死。RC3 的切回闪退率接近 100%。RC4 改为首次失去焦点后本次进程永久停止深度拦截。RC5 有所改善但仍不足；进一步确认正常运行时 Shader 加载不受明显影响，只有 Alt+Tab 切回触发的 ReShade 重载被插件拖慢。RC6 在整个 ReShade 效果重载窗口内暂停所有深度拦截。

当前版本号：`1.1-rc17` 顶点缓冲事件隔离版。RC14 不注册事件时正常，RC15 恢复生命周期回调后正常，RC16 增加五类被动回调后所有 Alt+Tab/Esc 问题复现。官方 ReShade 6.7.3 D3D9 源码显示，注册 `bind_vertex_buffers` 会启用 `DrawPrimitiveUP` 伪缓冲；Reset 释放这些缓冲后未清零句柄，后续 UP draw 会在失效句柄上调用 `get_resource_desc`，与固定崩溃偏移完全吻合。RC17 仅从 RC16 移除此事件，其余被动回调全部保留。

已验证二进制：`build/WeaponDepthMerge.addon32`。

版本隔离：`release/1.0` 分支和 `v1.0` 标签冻结旧基线；当前修复开发位于 `release/1.1`。`releases/1.0/` 保存 1.0 DLL 和 SHA-256，不得被 1.1 构建覆盖。

## 4. 原问题与 Thalixte 原理

Battlefield 2 将场景、第一人称武器、第一人称手臂分开渲染。武器/手臂阶段会清除或切换深度，因此普通 ReShade 最终读取到的深度无法同时包含场景和第一人称模型，AO 会穿过手臂。

Thalixte 的旧实现是 ReShade 核心内置功能 `Fix for user weapon or cockpit`，配置字段 `_brute_force_fix`。在普通 `DrawPrimitive` 和 `DrawIndexedPrimitive` 原始游戏 draw 之前，它会：

1. 判断当前是合适视口、选中的原始主深度源，并且已经跨过用户选定的 preserve clear；
2. 将 viewport 的 `MinZ`、`MaxZ` 同时减去 `0.5`；
3. 将可采样的 depth replacement 设为 DSV；
4. 将同一几何额外 draw 一次；
5. 恢复 viewport；
6. 随后游戏原 draw 正常发生。

旧版没有对 `DrawPrimitiveUP`/`DrawIndexedPrimitiveUP` 调用此功能。

重要限制：Thalixte 的额外 draw 没有关闭颜色写入。当前独立插件关闭颜色和 stencil 写入，只保留深度写入，避免双重颜色混合。这是独立插件相对于历史代码的有意安全改进。

## 5. 当前独立插件的实现

官方 Generic Depth 保持启用，用它把游戏的全尺寸主 D3D9 depth 替换成 INTZ，并提供 shader-resource usage。Generic Depth 的 `Copy depth buffer before clear operations` 必须关闭，因为它的 D3D9 preserve backup 是 `R32F copy_dest|shader_resource`，不能直接作为几何 DSV。

当前插件创建一张普通 D3D9 scratch depth，解决这个结构差异：

- INTZ：保留场景并接收额外的武器/手臂深度，最终供 ReShade `DEPTH` 使用；
- scratch depth：在选定 clear 时真正被清除，供游戏后续正常武器颜色 draw 使用。

选定 clear 后，每个普通 draw 的流程：

1. 保存 viewport、四个 color-write mask、stencil-write mask；
2. 绑定合并 INTZ；
3. 使用 `MinZ/MaxZ - DepthBias`，默认 `0.5`；
4. 禁用颜色和 stencil 写入，raw replay 一次，只写深度；
5. 恢复 viewport 和写入掩码；
6. 绑定 scratch depth，raw replay 一次，执行游戏正常颜色和局部遮挡；
7. 重新绑定合并 INTZ；
8. draw 事件返回 `true`，跳过原始 wrapper draw，因为两次所需 draw 已经执行。

ReShade 开始效果渲染时，插件解绑 DSV，并把合并 INTZ SRV 绑定到 `DEPTH` semantic。

## 6. 已经确认的官方 API 事实

这些事实已经从随仓库保存的官方 ReShade 6.7.3 源码核对，不应重复猜测：

- D3D9 `command_list::draw` 和 `draw_indexed` 直接调用底层 `_orig`，不会递归触发 add-on draw event。
- draw event 返回 `true` 会阻止 ReShade wrapper 再执行原始游戏 draw。
- `clear_depth_stencil_view` 返回 `true` 会阻止对应 depth/stencil clear 标志继续传给游戏 clear。
- `device->get_native()` 在 D3D9 返回底层 `IDirect3DDevice9 *`。
- Generic Depth 在 D3D9 `create_resource` 事件中将合适的主 depth 替换成 INTZ，并增加 shader-resource usage。
- D3D9 `bind_render_targets_and_depth_stencil(0, nullptr)` 保留颜色 RT，只解绑 DSV。
- Generic Depth 在不使用 backup texture 时也会在效果开始前解绑 DSV，以便采样同一 depth texture。
- D3D9 UP draw 在 add-on 事件前绑定 ReShade 的临时假 VB；通过比较事件 buffer 与真实 stream 0 可以识别并放行 UP draw。

## 8. 已知限制与风险

1. 仅支持 D3D9 x86。
2. 不支持 MSAA。
3. UP draws 不合并，这是历史行为兼容而非遗漏。
4. clear index 是启发式；换游戏或特殊场景可能需要人工选择。
5. selected boundary 后所有符合条件的普通 draws 都视为第一人称模型，可能包含 HUD 或后处理几何。
6. scratch depth 依次尝试 D24S8、D24X8、D16；极少数 RT/depth 格式组合可能不兼容。
7. 第一版只保存和恢复 viewport、颜色写 mask 和 stencil write mask；pipeline 其他状态不主动变化。
8. 双重 draw 可能影响 occlusion query 统计。Battlefield 2 当前实测未发现问题，但尚未做通用解决。
9. 事件加载顺序可能使 Generic Depth 再次更新 `DEPTH`；在 preserve-copy 关闭时两者应指向同一 INTZ，当前实测正常。
10. 目前没有自动化图像对比测试，最终正确性依赖游戏实测。
11. 1.1 只保证 DXVK/Vulkan 路径安全 no-op，不在 Vulkan 上实现深度合并。

### WDM-001：DXVK/Vulkan 启动崩溃

1.0 的 `draw`、`draw_indexed` 和 `clear_depth_stencil_view` 回调直接解引用 D3D9 私有状态。DXVK 下 ReShade 使用 Vulkan，插件不会创建 D3D9 状态，因此首次事件可能空指针退出。1.1 在所有跨 API 事件入口检查设备 API 和私有状态；非 D3D9 路径立即返回，不影响游戏。

RC1 已由项目所有者实测确认：DXVK 下功能不生效，但游戏可以正常进入，不再启动闪退。

### WDM-002：Alt+Tab 返回后画面变暗、卡死并退出

该问题从 1.0 就存在，RC1 仍受影响，原生 D3D9 和测试使用的 DXVK 配置均可出现。Alt+Tab 会触发 D3D9 丢失设备/重置；旧代码没有检查 `TestCooperativeLevel`，也假设 viewport、render state、深度表面切换、clear 和恢复操作一定成功。读取失败留下的零值可能被写回，且 scratch 深度切换失败时可能吞掉游戏原始颜色 draw。

RC2 在任何原生 D3D9 状态操作失败时立即停止拦截，实测已消除画面变暗，但卡死仍然发生。RC3 尝试前台暂停、延迟恢复和真实 DSV 恢复，实测切回闪退率接近 100%，该方案废弃。RC4 首次检测到进程失焦后永久设置 `focus_paused`，本次进程不再恢复拦截；同时不再动态增删回调，并在销毁资源视图时避免 combined SRV 重复释放。RC5 进一步修复 `on_begin_effects` 每帧调用 `update_texture_bindings("DEPTH", ...)` 的问题。RC6 在 `reshade_reloaded_effects` 到下一次 `reshade_begin_effects` 之间设置 `effects_reload_pending`，draw、clear 和绑定更新全部放行/暂停，直到 Shader 重载完成。

## 9. 下一步建议，按优先级排序

### P0：冻结成功基线

- 不要立即重构核心 draw/clear 状态机。
- 首先保存当前 DLL、SHA-256、ReShade 配置和成功截图。
- 后续每次修改前后运行 `scripts/verify-binary.ps1` 并在 Battlefield 2 做最小回归。

### P1：增强可诊断性

- 增加每帧 merged/failed/skipped 计数，而不是只显示累计值。
- 显示实际选中 clear、weapon phase 状态和 scratch depth 格式。
- 可选地记录 selected INTZ resource 变化和 device reset 次数。

### P2：覆盖游戏状态

- 测试换图、分辨率切换、全屏切换、死亡重生、载具座位、瞄具、菜单和 loading。
- 重点观察 D3D9 reset 后 `Combined INTZ` 是否能重新选择。

### P3：通用化

- 在保持 Battlefield 2 默认行为不变的前提下，增加 per-game preset。
- 评估按 vertex shader/pixel shader、viewport 或 render state 缩小 weapon phase draw 范围。
- MSAA 支持需要新的可绘制/可解析深度设计，不能简单打开开关。
